Verdict

Approve with suggestions.

The implementation is clean, well-layered, and has good test coverage. Two items worth addressing before merge: the deleted direct tests for GetComponentBySlug/GetSubComponentBySlug (loss of edge-case coverage), and calling h.config() multiple times per request (race with config reload). Neither is blocking, but both are easy fixes that strengthen the PR.

What this PR does

  • Adds GET /api/outages/during, a public endpoint that returns outages overlapping a time window or instant. Accepts start/end (RFC3339 or RFC3339Nano), plus optional componentName, subComponentName, tag, team filters.
  • Extracts SubComponentRefsMatching() on DashboardConfig to share AND-filtered sub-component lookup between ListSubComponentsJSON and the new endpoint.
  • Moves determineStatusFromSeverity from the handler to types.StatusFromOutages for reuse.
  • Adds ParseRFC3339OrNanoUTC and OutagesDuringQueryBounds utilities in pkg/utils.
  • Includes unit tests, handler-level tests, and e2e tests. Separately fixes MCP venv creation reliability.

Findings

Should fix

[should-fix] Deleted tests for GetComponentBySlug and GetSubComponentBySlug

pkg/types/config_test.go (entire file rewritten)

The old test file had direct tests for GetComponentBySlug and GetSubComponentBySlug covering edge cases: empty list, slug not found, case sensitivity (name vs slug). These were removed and replaced with tests only for SubComponentRefsMatching. The lookup methods are still public API and used in handler logic. The new tests cover some of this indirectly, but the dedicated edge-case coverage is gone. These were small, fast tests — worth keeping alongside the new ones.

[should-fix] Multiple h.config() calls per request risk config-reload inconsistency

cmd/dashboard/handlers.go:658-675 (ListSubComponentsJSON) and cmd/dashboard/handlers.go:681-736 (GetOutagesDuringJSON)

Both handlers call h.config() once for SubComponentRefsMatching, then again (implicitly via GetComponentBySlug/GetSubComponentBySlug) for each ref in the loop. If config reloads between these calls, refs may point to slugs that no longer exist. The nil guards prevent crashes, but items could be silently dropped. Fix: assign cfg := h.config() once at handler entry and use it throughout.

// Current (handlers.go:663-674):
refs := h.config().SubComponentRefsMatching(componentSlug, "", tag, team)
for _, ref := range refs {
    component := h.config().GetComponentBySlug(ref.ComponentSlug)  // second call
    ...
    sub := component.GetSubComponentBySlug(ref.SubSlug)

// Suggested:
cfg := h.config()
refs := cfg.SubComponentRefsMatching(componentSlug, "", tag, team)
for _, ref := range refs {
    component := cfg.GetComponentBySlug(ref.ComponentSlug)
    ...

Nits

[nit] Unreachable test case name is misleading

pkg/utils/query_time_test.go:76-78

The test case empty_both_hits_end_branch passes empty strings for both start and end. The handler rejects this before calling OutagesDuringQueryBounds, so this code path is unreachable in production. The test still exercises ParseRFC3339OrNanoUTC("") returning an error, which is fine, but the name suggests it's testing a real scenario. Consider renaming to something like empty_strings_returns_error.

Questions

[question] Unbounded query window on a public endpoint

cmd/dashboard/handlers.go:708-727, pkg/repositories/outage_repository.go:163-183

There's no limit on the time window size or the number of sub-component refs passed to the DB query. A caller could request start=2000-01-01&end=2099-01-01 with no component filter, hitting the full outages table across all sub-components. For a public endpoint, is this acceptable? Even a generous cap (e.g., 1 year, or a LIMIT on results) would prevent accidental or intentional abuse. This may be fine if the outages table is expected to stay small, but worth a conscious decision.

Things I checked and found fine

  • SQL safety: GetOutagesDuring builds the WHERE clause with ? placeholders only; user values go through GORM's parameterized args. The refs are pre-validated against config slugs. No injection risk.
  • Route ordering: /api/outages/during is registered before parameterized /api/components/{componentName}/... routes, so gorilla/mux won't shadow it.
  • Overlap query logic: start_time <= queryEnd AND (end_time IS NULL OR end_time > queryStart) correctly handles open-ended outages (NULL end_time) and point-in-time queries (queryStart == queryEnd).
  • StatusFromOutages: Identical logic to the deleted determineStatusFromSeverity, just relocated. Tests moved too and match 1:1.
  • SubComponentRefsMatching: AND semantics are correct and consistent with the existing ListSubComponentsJSON behavior before the refactor.
  • E2e tests: Test creates an outage, queries at mid-point (instant), queries with a range window, queries a non-overlapping window, and verifies 400 on missing params. Also tests tag+team filtering.
  • MCP venv fix (separate commit): Replaces --clear with rm -rf + fresh venv, and adds chmod -R u+w before removal. Reasonable fix for stale/broken venvs.

Open questions for the author

  • Is the outages table expected to stay small enough that an unbounded time window + all sub-components is fine, or should we add a reasonable cap (max window duration or result LIMIT)?
  • Was the removal of GetComponentBySlug/GetSubComponentBySlug tests intentional, or an oversight during the config_test.go rewrite? Happy to see them restored alongside the new SubComponentRefsMatching tests.
